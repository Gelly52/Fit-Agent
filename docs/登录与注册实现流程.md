# 后端登录流程原理

## 6 步运行流程
1. 用户在登录页先输手机号，请求 `/user/code`。后端先校验手机号格式，再生成 6 位验证码写入 Redis，并设置 5 分钟过期，接口本身不把验证码明文返回前端。
2. 用户再请求 `/user/login` 提交手机号和验证码。后端从 Redis 取出缓存验证码做比对，错误或过期直接返回失败；校验通过后立刻删除缓存，避免同一验证码被重复使用。
3. 验证码通过后，后端按手机号查询 `t_user`。如果数据库里没有这个手机号，就自动创建新用户，并生成稳定的 `userKey`，后续聊天、训练等业务都围绕这个身份做关联。
4. 登录成功后，后端生成随机 token，但数据库不保存明文 token，而是保存其 `sha256`。同时把用户ID、过期时间、IP、User-Agent 等信息写入 `t_user_login_session`，形成一条可追踪的登录会话记录。
5. 用户后续访问受保护接口时，统一先经过拦截器。拦截器从请求头 `headerUserToken` 取 token，做 hash 后查会话表，再校验会话是否过期、是否已注销、用户是否被禁用，通过后才放行业务逻辑。
6. 用户退出时调用 `/user/logout`。后端找到当前 token 对应的会话记录，把 `revokedAt` 设置为当前时间。这样旧 token 即使还留在浏览器里，服务端也会把它当作失效会话，后续请求直接拦截。

## 对应代码实现逻辑
1. `UserController.sendCode` 接收 `UserCodeRequest`，再调 `UserServiceImpl.sendCode`。服务层先执行 `validatePhone` 校验手机号，然后用 `RandomUtil.randomNumbers(6)` 生成验证码，再通过 `StringRedisTemplate` 以 `fit-agent:login:code:手机号` 为键写入 Redis。

2. `UserController.login` 会把手机号、验证码、客户端 IP、User-Agent 传给 `UserServiceImpl.login`。登录方法先拼出 Redis key，再读取缓存验证码做比较；只要为空、过期或不一致，就直接抛出“验证码错误或已过期”。

3. 验证码校验成功后，代码会立刻删除 Redis 中的验证码。接着调用 `queryUserByPhone` 查库；如果用户不存在，就走 `createUserWithPhone` 创建用户。这里会生成 `u_` 开头的 `userKey`，并用唯一键异常兜底并发注册。

4. 会话创建发生在 `UserServiceImpl.login` 后半段：先更新用户 `lastLoginAt`，再用 `IdUtil.fastSimpleUUID()` 生成 token，用 `SecureUtil.sha256(token)` 存到 `UserLoginSession.refreshTokenHash`。最后通过 `buildLoginResponse` 把 token、过期时间和用户信息返回前端。

5. 统一鉴权由 `AuthWebMvcConfig` 和 `TokenAuthInterceptor` 完成。前者把 `/user/**`、`/chat/**`、`/rag/**`、`/internet/**`、`/sse/**` 纳入拦截，只放行 `/user/code`、`/user/login`、`/sse/connect` 等白名单；后者只认 `headerUserToken`，校验成功后把用户上下文写入 `request` 和 `ThreadLocal`。

6. 退出登录在 `UserController.logout` 和 `UserServiceImpl.logout` 中完成。控制器从请求头取 `headerUserToken`，服务层按 token 的 hash 查询未注销会话，找到后把 `revokedAt` 更新为当前时间。之后旧 token 再进拦截器时，就会因为会话已失效被拒绝。
