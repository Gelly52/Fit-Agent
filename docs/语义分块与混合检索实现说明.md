# 语义分块与混合检索实现说明

## 1. 文档说明

本文档说明当前项目代码中已经具备的语义分块与混合检索能力，重点覆盖：

- 文档上传后的分块与入库流程
- 语义分块算法的主要规则
- 向量检索、关键词检索与 RRF 融合流程
- 当前开发配置中实际生效的参数
- 对外接口与联调关注点

本文档以当前代码和当前开发配置为准，主要依据以下文件：

- `Fit-Agent-backend/mcp-client/src/main/java/com/itgeo/service/impl/DocumentServiceImpl.java`
- `Fit-Agent-backend/mcp-client/src/main/java/com/itgeo/service/chunking/SemanticDocumentChunker.java`
- `Fit-Agent-backend/mcp-client/src/main/java/com/itgeo/service/impl/RedisKeywordSearchServiceImpl.java`
- `Fit-Agent-backend/mcp-client/src/main/java/com/itgeo/service/impl/RrfRagFusionServiceImpl.java`
- `Fit-Agent-backend/mcp-client/src/main/resources/application-dev.yml`

## 2. 当前能力概览

当前项目代码已经支持以下两类能力：

1. 语义分块能力
   - 代码中已经实现 `SemanticDocumentChunker`，支持按段落、句子和句向量相似度进行语义切分。
   - 当分块策略配置为 `semantic` 时，上传文档会走语义分块流程。

2. 混合检索能力
   - 检索链路中已经同时具备向量召回、关键词召回和 RRF 融合能力。
   - 当检索模式配置为 `hybrid` 时，会先分别召回向量结果和关键词结果，再进行融合排序。

需要注意的是：

- 当前开发配置 `application-dev.yml` 中，检索模式已经是 `hybrid`。
- 当前开发配置中的分块策略仍然是 `token`，不是 `semantic`。
- 也就是说，代码已经支持语义分块，但当前开发环境默认并未启用语义分块。

## 3. 当前开发配置

当前活动配置文件：`Fit-Agent-backend/mcp-client/src/main/resources/application-dev.yml`

### 3.1 Embedding 与向量索引

```yml
rag:
  embedding:
    provider: bge-m3-http
    service-url: http://127.0.0.1:7086
    model-name: bge-m3
    dimension: 1024

  vectorstore:
    index-name: lee-vectorstore-bgem3
    prefix: "embedding:bgem3:"
```

含义：

- 当前 embedding 提供方使用本地 HTTP 服务 `bge-m3-http`
- 当前向量索引名称为 `lee-vectorstore-bgem3`
- 当前向量 key 前缀为 `embedding:bgem3:`

### 3.2 混合检索参数

```yml
rag:
  retrieval:
    mode: hybrid
    default-top-k: 4
    max-top-k: 10
    vector-recall-k: 16
    keyword-recall-k: 16
    rrf-k: 60
    vector-weight: 1.0
    keyword-weight: 1.0
```

含义：

- 当前检索模式为 `hybrid`
- 最终默认返回数量是 `4`
- 向量召回候选数是 `16`
- 关键词召回候选数是 `16`
- 融合算法使用 RRF，`rrf-k = 60`
- 当前向量和关键词的默认权重相同，均为 `1.0`

### 3.3 分块参数

```yml
rag:
  chunking:
    strategy: token
    merge-threshold: 0.20
    breakpoint-drop-threshold: 0.55
    max-chunk-sentence-count: 80
    max-chunk-chars: 1200
    sentence-embedding-batch-size: 64
    paragraph-boundary-enabled: false
    debug-log-enabled: false
```

含义：

- 当前开发配置中的分块策略是 `token`
- 因此上传文档时默认会走 `TokenTextSplitter` 风格的切分逻辑
- `merge-threshold` 等参数是为语义分块准备的，当前配置下未直接生效

## 4. 文档上传与分块入库流程

上传入口是：`POST /rag/uploadRagDoc`

主流程位于：`DocumentServiceImpl.loadText(...)`

当前流程如下：

1. 校验 `userId` 非空
2. 使用 `TextReader` 读取原始文本
3. 把 `fileName` 和 `userId` 写入原始文档 metadata
4. 调用 `semanticDocumentChunker.splitDocuments(documentList)` 生成 chunk 列表
5. 为每个 chunk 补齐以下 metadata：
   - `fileName`
   - `userId`
   - `source`
   - `documentId`
   - `chunkId`
   - `chunkSeq`
6. 调用 `redisVectorStore.add(splitDocuments)` 写入向量库
7. 调用 `keywordSearchService.indexChunks(splitDocuments)` 建立关键词索引数据
8. 把文档索引信息写入 `t_rag_document`

说明：

- 当前上传链路不是只做向量入库，也会同步建立关键词检索所需的数据。
- 因此混合检索的两路召回，都是基于同一批 chunk 元数据构建的。

## 5. 语义分块实现说明

语义分块实现位于：`SemanticDocumentChunker`

### 5.1 分块入口

`splitDocuments(...)` 会先检查配置：

- 如果 `chunking.strategy != semantic`
  - 直接回退到 `TokenTextSplitter`
- 如果 `chunking.strategy == semantic`
  - 进入语义分块主流程

这也是为什么当前开发配置虽然已经具备语义分块代码，但默认仍然走 token 分块。

### 5.2 语义分块主流程

当启用 `semantic` 时，核心流程如下：

1. 先按段落边界切分文本
   - 如果 `paragraph-boundary-enabled = true`，空行会被视为段落边界
   - 如果该开关关闭，则整个文本视为一个段落

2. 再按句子切分段落
   - 目前按中英文常见句末标点切句，例如 `。！？；.!?;`

3. 对句子列表批量计算 embedding
   - 批大小由 `sentence-embedding-batch-size` 控制

4. 基于相邻句向量相似度决定是否继续合并
   - 如果当前相似度低于 `merge-threshold`，则断开
   - 如果相似度相较上一句对出现明显突降，且超过 `breakpoint-drop-threshold`，则断开
   - 如果达到 `max-chunk-sentence-count`，则断开
   - 如果预计字符数超过 `max-chunk-chars`，则断开

5. 对最终得到的语义 chunk 执行 overflow fallback
   - 如果某个语义 chunk 仍然过长，则继续回退到 `TokenTextSplitter` 拆分

### 5.3 语义分块的目标

语义分块的目的不是机械按固定长度切片，而是尽量让一个 chunk 内部保持主题连续性，从而提升：

- 向量召回时的语义命中质量
- 知识问答时的上下文完整性
- benchmark 场景下的文件级命中率稳定性

## 6. 混合检索实现说明

混合检索的主入口位于：`DocumentServiceImpl.doSearchWithTrace(...)`

### 6.1 检索主流程

当前流程如下：

1. 校验 `userId` 和 `question`
2. 根据请求值规范化 `finalTopK`
3. 读取配置中的：
   - `vectorRecallK`
   - `keywordRecallK`
4. 执行向量召回：`vectorRecall(question, userId, vectorRecallK)`
5. 执行关键词召回：`keywordSearchService.search(question, userId, keywordRecallK)`
6. 执行融合：`ragFusionService.fuse(vectorHits, keywordHits, finalTopK)`
7. 将融合结果映射为最终文档列表返回

### 6.2 向量检索

向量检索当前通过 `RedisVectorStore.similaritySearch(...)` 完成。

关键点：

- 检索时使用 `filterExpression("userId == '...'")` 做用户隔离
- 这意味着向量召回阶段就会把不属于当前用户的数据过滤掉
- 当前不是“先查全量，再在 Java 内存中过滤”的实现

### 6.3 关键词检索

关键词检索当前由 `RedisKeywordSearchServiceImpl` 实现。

当前做法：

1. 上传时把 chunk 内容及 metadata 写入 Redis Hash
2. 查询时通过 RediSearch 执行：
   - `(@userId:{当前用户}) 问题文本`
3. 返回的结果会转换为 `RagRetrievedChunk`

因此关键词检索和向量检索一样，也基于 `userId` 做用户范围约束。

### 6.4 RRF 融合

融合实现位于：`RrfRagFusionServiceImpl`

当前采用的是 RRF（Reciprocal Rank Fusion）策略：

- 向量结果按 `vectorWeight / (rrfK + rank)` 计分
- 关键词结果按 `keywordWeight / (rrfK + rank)` 计分
- 若同一个 `chunkId` 同时出现在两路结果中，会累加分数
- 最后按融合分数降序排序，并截断到最终 `topK`

当前开发配置中：

- `vectorWeight = 1.0`
- `keywordWeight = 1.0`

因此默认是等权融合。

## 7. 对外接口

当前与语义分块、混合检索直接相关的接口主要有：

### 7.1 上传文档

- `POST /rag/uploadRagDoc`

作用：

- 读取文本
- 分块
- 写向量索引
- 写关键词索引
- 落库文档索引记录

### 7.2 手动检索

- `GET /rag/doSearch`

作用：

- 对当前登录用户执行检索
- 当前控制器固定使用 `topK = 4`
- 返回融合后的最终文档列表

### 7.3 手动 RAG 问答

- `POST /rag/search`

作用：

- 先检索，再把检索结果交给聊天服务生成回答

### 7.4 读取 RAG 配置

- `GET /rag/config`
- `POST /rag/config`

作用：

- 返回当前代码中生效的手动 RAG 配置快照
- 便于前端展示与联调排查

## 8. 当前实现边界

当前代码虽然已经支持语义分块与混合检索，但仍有一些边界需要明确：

1. 当前开发配置默认启用的是 `hybrid`，但分块策略仍是 `token`
2. 语义分块参数虽然已经存在，但只有在 `chunking.strategy = semantic` 时才会真正生效
3. 当前 RAG 检索范围仍然严格按登录用户隔离
4. 当前 benchmark 评估的是检索命中率，不直接评估答案正确率
5. 当前 `/rag/config` 返回的是代码侧配置快照，不是数据库动态配置

## 9. 使用建议

如果你希望实际启用“语义分块 + 混合检索”的组合，可以重点关注：

1. 在配置中把 `rag.chunking.strategy` 切到 `semantic`
2. 重新上传文档，让新的 chunk 重新入向量库和关键词索引
3. 结合 benchmark 观察：
   - `vectorRecallK`
   - `keywordRecallK`
   - `vectorWeight`
   - `keywordWeight`
   - `rrfK`
4. 根据数据集特点调整语义分块参数，例如：
   - `merge-threshold`
   - `breakpoint-drop-threshold`
   - `max-chunk-sentence-count`
   - `max-chunk-chars`
